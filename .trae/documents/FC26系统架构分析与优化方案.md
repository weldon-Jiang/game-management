## 二、详细问题分析

### 问题 1: JWT Secret 密钥处理

**当前代码:**
```java
private SecretKey getSigningKey() {
    byte[] keyBytes = secret.getBytes(StandardCharsets.UTF_8);
    byte[] paddedKey = new byte[32];
    System.arraycopy(keyBytes, 0, paddedKey, 0, Math.min(keyBytes.length, 32));
    return Keys.hmacShaKeyFor(paddedKey);
}
```

**问题:**
- 如果 secret 长度小于 32 字节，会用零填充，可能导致弱密钥
- 如果 secret 长度大于 32 字节，会被截断

**修复方案:**
```java
private SecretKey getSigningKey() {
    byte[] keyBytes = secret.getBytes(StandardCharsets.UTF_8);
    if (keyBytes.length < 32) {
        throw new IllegalStateException("JWT secret must be at least 32 characters");
    }
    return Keys.hmacShaKeyFor(keyBytes);
}
```

### 问题 2: 密钥比较时序攻击

**当前代码:**
```java
return agentSecret.equals(instance.getAgentSecret());
```

**问题:**
- String.equals() 不是常量时间比较
- 攻击者可以通过测量响应时间猜测密钥

**修复方案:**
```java
import java.security.MessageDigest;

public static boolean constantTimeEquals(String a, String b) {
    if (a == null || b == null) {
        return false;
    }
    return MessageDigest.isEqual(a.getBytes(), b.getBytes());
}
```

### 问题 3: refreshToken 过期时间未更新

**当前代码:**
```java
public String refreshToken(String token) {
    Claims claims = parseToken(token);
    // ... 提取 claims
    return generateToken(userId, username, merchantId, role); // 过期时间未更新
}
```

**问题:**
- refreshToken 重新生成 token 但过期时间沿用旧的
- 导致 refresh 后的 token 实际过期时间比预期早

**修复方案:**
```java
public String refreshToken(String token) {
    Claims claims = parseToken(token);
    String userId = claims.get("userId", String.class);
    String merchantId = claims.get("merchantId", String.class);
    String role = claims.get("role", String.class);
    String username = claims.getSubject();
    return generateToken(userId, username, merchantId, role); // 已修复，会生成新的过期时间
}
```

### 问题 4: Task merchantId 可能为空

**位置:** `TaskController.create` 等

**问题:** 创建 Task 时可能未强制设置 merchantId

**修复方案:**
- 在 Service 层创建 Task 时强制从 UserContext 获取 merchantId
- 添加校验，merchantId 为空时拒绝创建

// MerchantBalance 添加 version 字段

## 三、优化实施计划

### 第一阶段：安全修复 (Critical)

1. **JwtUtil 安全修复**
   - 添加密钥长度校验
   - 确保 refreshToken 生成新的过期时间

2. **时序攻击修复**
   - 将 equals 改为常量时间比较

3. **敏感字段脱敏**
   - 在 DTO 中过滤密码字段

### 第二阶段：数据一致性 (High)

4. **Task merchantId 强制校验**
   - 在 TaskService.create 时校验

5. **MerchantBalance 乐观锁**
   - 添加 @Version 注解

6. **Agent 心跳并发优化**
   - 使用数据库更新代替先查后改

### 第三阶段：性能优化 (Medium)

7. **批量更新优化**
   - offlineByTimeout 使用批量 SQL

8. **API 规范化**
   - 统一 RESTful 风格
