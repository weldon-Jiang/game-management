#### 2.1.3 充值系统推荐方案

**推荐: 方案C - 融合方案（阶梯充值卡 + VIP等级折扣）**

理由：
- B端工作室注重实际折扣，阶梯充值直接让利
- VIP等级鼓励长期合作，提升粘性
- 卡密系统便于分销和子账户管理

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           融合方案实施细则                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【阶梯充值卡】面额与折扣                                                     │
│  ├── 100元  → 100点 (原价，无折扣)                                           │
│  ├── 500元  → 500点 (980点，98折)                                           │
│  ├── 1000元 → 1000点 (975点，975折) ← 推荐                                   │
│  ├── 5000元 → 5000点 (4600点，92折)                                         │
│  └── 10000元→ 10000点(9000点，90折)                                         │
│                                                                             │
│  【VIP等级折扣】累计充值                                                    │
│  ├── VIP1: 累计0元    → 折扣: 100% (原价)                                   │
│  ├── VIP2: 累计5000元 → 折扣: 95折                                          │
│  ├── VIP3: 累计20000元→ 折扣: 90折                                          │
│  └── VIP4: 累计50000元→ 折扣: 85折                                          │
│                                                                             │
│  【最终点数计算】                                                           │
│  最终点数 = 购买点数 × VIP折扣率                                             │
│  例: VIP3用户购买1000元卡 → 975点 × 0.90 = 877.5 ≈ 878点                   │
│                                                                             │
│  【卡密 vs 直接授权】                                                       │
│  ├── 卡密: 可分发子账户，防作弊，适合分销                                     │
│  └── 直接授权: 便捷，内部使用                                                 │
│  → 推荐: 主要用卡密，辅助直接授权                                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.3 绑定与解绑策略（P1）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           设备绑定/解绑流程                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【绑定流程】                                                               │
│  1. 用户申请绑定设备 → 检查点数是否充足                                       │
│  2. 验证设备有效性 → 查询是否已被绑定                                         │
│  3. 扣除点数 → 创建device_binding记录                                        │
│  4. 开通服务 → 创建subscription记录                                         │
│                                                                             │
│  【解绑流程】                                                               │
│  1. 用户申请解绑 → 检查7天内解绑次数(<max_unbind_per_week)                   │
│  2. 计算返还点数 = 剩余天数/总天数 × 原价 × refund_rate                        │
│  3. 更新device_binding状态 → 更新subscription状态                           │
│  4. 记录point_transaction(返还)                                             │
│                                                                             │
│  【防滥用机制】                                                             │
│  ├── 7天限制解绑次数: max_unbind_per_week                                  │
│  ├── 解绑返还比例可配置: unbind_refund_rate (默认50%)                        │
│  └── 解绑需审核: 可配置是否需要人工审核                                      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 五、风险评估与规避

| 风险 | 影响等级 | 规避方案 |
|------|---------|---------|
| **PS5串流延迟高导致自动化不稳定** | 🔴 高 | Chiaki-NG + 低码率模式 + 帧预测补偿 |

| **游戏版本更新导致自动化失效** | 🟡 中 | 模块化设计 + 快速适配版本号检测 + 模板更新机制 |
| **EA反作弊检测** | 🟡 中 | 随机化操作间隔 + 模拟人工操作模式 |
| **客户恶意刷解绑** | 🟢 低 | 7天限制 + 人工审核开关 + 返还比例可配置 |
